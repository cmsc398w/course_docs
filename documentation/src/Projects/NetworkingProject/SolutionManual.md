# Solution Manual

## Ticket 1

We had connectivity issues between containers due to nginx's binding configuration in the crm-server container. When attempting to connect from the sales-client container using `curl 10.0.0.50:8080`, the connection was refused despite both containers being on the same Docker network. Using the ss tool's socket analysis capabilities with `ss -tlnp | grep nginx` revealed that nginx was only listening on `127.0.0.1:8080`, explaining why communication failed since localhost binding doesn't accept external connections. The fix involved modifying the nginx configuration from `listen 127.0.0.1:8080 default_server;` to `listen 8080 default_server;`, allowing the server to accept connections on all interfaces. After rebuilding and restarting the containers, the connection from sales-client to crm-server was successfully established, confirming the resolution of the binding issue.

## Ticket 2

The issue affects the company blog's DNS resolution because the client container is configured to use an incorrect DNS server (10.0.0.54) that isn't responding to queries. Running `dig @10.0.0.54 blog.company.internal` from the client container returns a "connection refused" error, while `dig @10.0.0.53 blog.company.internal` successfully resolves to the blog server's IP (10.0.0.100), confirming that the DNS resolution is failing due to misconfiguration rather than a network connectivity issue. This can be diagnosed using `cat /etc/resolv.conf` to confirm the client is pointing to the wrong nameserver, and verified with tcpdump using `tcpdump -n port 53` to observe the DNS queries being sent to the non-functional server. The solution involves updating the client's DNS configuration in `/etc/resolv.conf` to use the correct DNS server at 10.0.0.53, after which DNS resolution for blog.company.internal and other internal domains will work properly.

## Ticket 3

The issue is that there is no route to the 10.0.0.0/24 network from where the client is. Running `ip route get 10.0.0.75` from the client container returns "Network is unreachable", while `ip route show`  hasno routing entry exists to reach the API server's 10.0.0.0/24 network from the client's location in 10.1.0.0/24. We can further verify with `traceroute 10.0.0.75`, which fails to find any path to the destination. The solution requires establishing network connectivity between the two segments by adding an appropriate route via `ip route add 10.0.0.0/24 via [gateway-ip]`.
