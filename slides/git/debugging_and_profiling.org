* Diagnostic Mindset

** Why does this matter?
:PROPERTIES:
:BEAMER_ENV: block
:END:

Software does exactly what you tell it, not what you intend.

- Debugging = bridging the gap between intended vs. actual behavior
- Structured debugging is methodical, not trial-and-error
- Avoid just changing things randomly until it works

** The Scientific Method for Debugging
:PROPERTIES:
:BEAMER_ENV: exampleblock
:END:

#+BEGIN_EXPORT latex
\begin{center}
\begin{tikzpicture}[node distance=2.5cm, auto,
    block/.style={rectangle, draw, fill=blue!20, text width=5em, text centered, rounded corners, minimum height=3em},
    line/.style={draw, -latex', thick}]
    \node [block] (repro) {Reproduce};
    \node [block, right of=repro] (iso) {Isolate};
    \node [block, right of=iso] (hyp) {Hypothesize};
    \node [block, right of=hyp] (test) {Test};
    \node [block, below of=iso] (fix) {Fix \& Verify};

    \path [line] (repro) -- (iso);
    \path [line] (iso) -- (hyp);
    \path [line] (hyp) -- (test);
    \path [line] (test) -- (fix);
    \path [line] (test) edge[bend right=45] node[above] {Fail} (hyp);
\end{tikzpicture}
\end{center}
#+END_EXPORT

1. *Reproduce* - If you can't reproduce, you can't fix
2. *Isolate* - Binary search through code
3. *Hypothesize* - Form theory based on symptoms
4. *Test* - Change ONE thing at a time
5. *Fix & Verify* - Confirm it solves the issue

** Real-World Example
:PROPERTIES:
:BEAMER_ENV: block
:END:

In 2020, Discord saved millions in hardware costs by profiling their "Read States" service (tracking which messages you’ve read).

* Symptom: Every 2 minutes, the service experienced massive latency spikes and irregular CPU usage despite no change in user traffic.
* Debugging / Profiling: Developers used flame graphs and heap profilers to realize the Garbage Collector was trying to scan a cache of items.
* Discovery: Every time the cache grew, the GC had to work harder to inspect  it, creating a pause that slowed down every user request.
* Result: They moved the service to **Rust** (which has no GC), reducing CPU usage by **10x** using the exact same hardware.


** Rubber Duck Debugging
:PROPERTIES:
:BEAMER_ENV: block
:END:

/The practice of explaining your code, line-by-line, to an inanimate object./

- Forces you to slow down and articulate your logic.
- Often, you find the bug before finishing the explanation.
- Works because it shifts brain from "creation" mode to "explanation" mode.

** Divide and Conquer
:PROPERTIES:
:BEAMER_ENV: exampleblock
:BEAMER_ACT: <2->
:END:

How to find a bug in 10,000 lines of code?

1. *Binary Search the Code*: Comment out half the code. Does it still crash?
   - Yes $\rightarrow$ Bug is in the active half.
   - No $\rightarrow$ Bug is in the commented half.
2. *Git Bisect*: Binary search through commit history to find *when* the bug was introduced.
   - Good commit vs. Bad commit.

* Printf Debugging & Logging

** Print Debugging: The Foundation
:PROPERTIES:
:BEAMER_ENV: block
:END:

Brian Kernighan (1979): /"The most effective debugging tool is still careful thought, coupled with judiciously placed print statements."/

- Print statements = simplest debugging method
- Iterate by adding prints near problem areas until root cause is clear
- Preferred because it's quick, easy, and effective

** Logging: Enhanced Print Debugging
:PROPERTIES:
:BEAMER_ENV: exampleblock
:BEAMER_ACT: <2->
:END:

Logging advantages:

- Output to files, sockets, remote servers (easier to review than terminal)
- Severity levels: =INFO=, =DEBUG=, =WARN=, =ERROR=
- Long-term monitoring + immediate debugging
- Structured logging (key–value pairs) $\rightarrow$ searchable, filterable, analyzable

** Logging Best Practices
:PROPERTIES:
:BEAMER_ENV: block
:END:

***                                                          :B_columns:

**** Best Practices                                           :BMCOL:

- Set log levels properly (filter noise)

* Specialized Tools

** System Call Tracing
:PROPERTIES:

:END:

Monitor interactions between a program and the OS kernel.

- Tools: =strace= (Linux), =dtruss= (macOS)
- Useful for debugging programs where you cannot see source code
- Shows every system call made by a program with arguments and results

* Profiling

** Beyond Functional Correctness
:PROPERTIES:
:BEAMER_ENV: block
:END:

Even if your code behaves correctly, it might not be good enough if it takes all your CPU or memory.

- Algorithms classes teach big $O$ notation but not how to find hot spots
- *Premature optimization is the root of all evil*
- Profilers help you understand which parts take most time/resources
- Focus optimization efforts where they matter most

* Profiling via Internal Timing

** Understanding Time Metrics
:PROPERTIES:
:BEAMER_ENV: block
:END:

***                                                          :B_columns:
:PROPERTIES:
:BEAMER_ENV: columns
:END:

**** Time Types                                               :BMCOL:
:PROPERTIES:
:BEAMER_COL: 0.48
:END:

| Type | What It Measures |
|------|------------------|
| *Real* | Wall clock time (total) |
| *User* | CPU time in your code |
| *Sys* | CPU time in OS calls |

Real = User + Sys + Wait Time

**** Key Insight                                              :BMCOL:
:PROPERTIES:
:BEAMER_COL: 0.48
:END:

If Real >> User + Sys:
- Program is *waiting* (I/O, network)
- Optimize by reducing waits
- Cache data, batch requests
- Make fewer external calls

If User is high:
- CPU-bound computation
- Optimize algorithms

* CPU Profilers

** CPU Profilers: Tracing vs Sampling
:PROPERTIES:
:BEAMER_ENV: block
:END:

Tools to analyze and measure CPU time spent on different functions/code blocks.

***                                                          :B_columns:
:PROPERTIES:
:BEAMER_ENV: columns
:END:

**** Tracing Profilers                                        :BMCOL:
:PROPERTIES:
:BEAMER_COL: 0.48
:END:

- Track *every* function call your program makes
- Complete record of execution
- Higher overhead
- Precise call counts

**** Sampling Profilers                                       :BMCOL:
:PROPERTIES:
:BEAMER_COL: 0.48
:END:

- Periodically sample program's stack (e.g., every millisecond)
- Aggregate statistics of time usage
- Lower overhead
- Statistical approximation

**** Common Memory Issues                                     :BMCOL:
:PROPERTIES:
:BEAMER_COL: 0.48
:END:

- *The Global List*: Cache that never clears
- *Unclosed Resources*: File/DB connections not closed
- *Reference Cycles*: Objects pointing to each other
- *Large Data Loading*: Reading 5GB files into RAM

**** Profiling Approaches                                     :BMCOL:
:PROPERTIES:
:BEAMER_COL: 0.48
:END:

- *Heap Profiling*: Track allocations/deallocations
- *Reference Counting*: Monitor object references
- Tools: =memory_profiler=, =tracemalloc=, =valgrind=

*** Key Strategy
:PROPERTIES:
:BEAMER_ENV: alertblock
:BEAMER_ACT: <2->
:END:

*Avoid unnecessary allocations* - they take time and trigger garbage collection. Reuse memory when possible, process large files line-by-line.

* Summary

** Choosing the Right Tool
:PROPERTIES:
:BEAMER_ENV: block
:END:

Different tools for different problems:

| Tool | Best For... | Trade-offs |
|------|-------------|------------|
| *Print/Logs* | Flow tracing, production debugging | Requires code changes, re-running |
| *Debugger* | Deep inspection, state modification | Manual, slower, hard for concurrency |
| *Profilers* | Finding performance bottlenecks | Doesn't fix correctness, adds overhead |
| *strace/dtruss* | System call issues, black-box debugging | Low-level, verbose output |
| *git bisect* | Finding when bug was introduced | Need reproducible test case |
