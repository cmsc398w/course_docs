* Introduction

** What is Git?

*** Definition
:PROPERTIES:
:BEAMER_ENV: block
:END:

- Git is a open-source distributed version control system (VCS) designed to track changes in files and source code during software development.
- Git != GitHub


*** History
:PROPERTIES:
:BEAMER_ENV: block
:END:

- Created by *Linus Torvalds* in 2005 for Linux kernel development.
- Built for speed and data integrity in a distributed environment.

*** Core Philosophy
:PROPERTIES:
:BEAMER_ENV: exampleblock
:BEAMER_ACT: <2->
:END:

Unlike SVN or CVS, Git gives every developer a *complete copy* of the entire history. It is a *distributed* version control system.

** Why Git?

*** The Problem
:PROPERTIES:
:BEAMER_ENV: block
:END:

- "It worked yesterday. What changed?"
- "I deleted that function and now I need it back."
- "My teammate's changes broke my code."

*** The Alternative (Don't do this!)
:PROPERTIES:
:BEAMER_ENV: alertblock
:BEAMER_ACT: <2->
:END:

- =project_v1.zip=, =project_final.zip=, =project_FINAL_FINAL.zip=
- Emailing code to yourself.

Git replaces this with a snapshot of your project at every commit.

* Viewing History

** Inspecting Your State

*** Commands
:PROPERTIES:

:END:

| Command | What It Shows |
|---------|---------------|
| =git log= | Full commit history |
| =git log --oneline --graph --all= | Compact graph view of all branches |
| =git diff= | Changes not yet staged |
| =git diff --staged= | Changes staged for the next commit |
| =git show HEAD= | Full detail of the last commit |

** Who Changed This?

*** git blame <file>
:PROPERTIES:
:BEAMER_ENV: block
:END:

Shows which user modified each line of a file, when they did it, and in which commit.

- Excellent for finding the *context* behind a line of code.
- Combine with =git show <hash>= to see the full change.

* Undoing Things

** Revert: Safe Undo

*** git revert <hash>
:PROPERTIES:
:BEAMER_ENV: block
:END:

Creates a *new commit* that is the inverse of the target commit. History is never rewritten, making it safe for shared branches.

** Reset: Undo Local Commits

*** git reset
:PROPERTIES:
:BEAMER_ENV: exampleblock
:END:

Moves the branch pointer to an earlier commit. Rewrites history; only use on unpushed commits.

| Mode | Command | Side Effect (all will undo commits) |
|------|---------|--------|
| Soft | =git reset --soft HEAD~1= | Changes stay *staged*              |
| Mixed (default) | =git reset HEAD~1= | Changes go to *working directory*              |
| Hard | =git reset --hard HEAD~1= | Changes are *discarded*              |

*** Relative Refs
:PROPERTIES:
:BEAMER_ENV: block
:END:

=HEAD~N= means N commits behind HEAD:

| Ref | Meaning |
|-----|---------|
| =HEAD~1= or =HEAD~= | One commit back (parent) |
| =HEAD~2= | Two commits back (grandparent) |

** The Reflog

*** Using the Reflog
:PROPERTIES:
:BEAMER_ENV: exampleblock
:END:

Git records *every time HEAD moves*, including destructive operations.

#+BEGIN_SRC bash
# Destroyed work with a hard reset
$ git reset --hard HEAD~2

# Find the lost commit
$ git reflog
a1b2c3d HEAD@{2}: commit: Add payment feature  <- this one!

# Restore it
$ git reset --hard a1b2c3d
#+END_SRC

** Discussion: Reset or Revert?

*** Scenario
:PROPERTIES:
:BEAMER_ENV: block
:END:

You pushed a commit to =main= that breaks the production server. Your team has already pulled it.

*** Question (Discuss at your tables)
:PROPERTIES:
:BEAMER_ENV: exampleblock
:END:

Do you use =git reset --hard= or =git revert=?
What happens to your teammates' history if you pick the wrong one?

*** Answer
:PROPERTIES:
:BEAMER_ENV: alertblock
:BEAMER_ACT: <2->
:END:

Use *=git revert=*. It creates a new "undo" commit that others can pull safely. Resetting rewrites history, which would cause your teammates' local repos to diverge and break their next pull.

** Stashing: Save for Later

*** The Scenario
:PROPERTIES:
:BEAMER_ENV: block
:END:

You're in the middle of a feature, but need to fix a bug on =main=. You don't want to commit half-broken work.

*** Commands
:PROPERTIES:
:BEAMER_ENV: exampleblock
:END:

| Command | Action |
|---------|--------|
| =git stash= | Saves all uncommitted changes and clears your WD |
| =git stash pop= | Restores stashed work and removes it from the stack |
| =git stash list= | View all stashed items |
| =git stash apply= | Restores stashed work but keeps it in the stack |

* Working with Remotes

** Remotes: Another Copy of the Repo

*** Overview
:PROPERTIES:
:BEAMER_ENV: block
:END:

A *remote* is another copy of the repository, on GitHub's servers or a teammate's machine. The graphs stay in sync by pushing and pulling commits.

| Term | Meaning |
|------|---------|
| =origin= | Default nickname for the remote URL |
| =upstream= | The original repo you forked from |
| =origin/main= | Local bookmark of where the remote's branch last was |

** Managing Remotes

*** Commands
:PROPERTIES:
:BEAMER_ENV: block
:END:

| Command | Action |
|---------|--------|
| =git remote add <name> <url>= | Connect local repo to a new server |
| =git remote -v= | List all configured remotes |
| =git remote remove <name>= | Delete a remote connection |

- =origin= is just the default name for the first remote.
- You can have multiple remotes (e.g., =origin=, =upstream=).

** Push, Fetch, Pull

*** Commands
:PROPERTIES:
:BEAMER_ENV: block
:END:

| Command | What It Does |
|---------|--------------|
| =git push <remote> <branch>= | Upload commits to a specific server/branch |
| =git push -u origin <branch>= | First push; links local branch to remote |
| =git fetch <remote>= | Download new commits; does not touch working files |
| =git pull <remote> <branch>= | =fetch= + =merge= in one step |

- Plain =git pull= only works if a branch is "tracking" a remote (set with =-u=).
- Use =git fetch= to see what changed before merging.

** Detached HEAD

*** Detached HEAD State
:PROPERTIES:
:BEAMER_ENV: alertblock
:END:

Checking out a raw commit hash (e.g. =git switch --detach a1b2c3d=) puts you in *detached HEAD* state.

- =HEAD= points to a commit, not a branch name.
- Any commits you make have no branch label and will be lost.
- Fix: run =git switch -c new-branch-name= immediately.

** Conflict Resolution

*** When Conflicts Happen
:PROPERTIES:
:BEAMER_ENV: block
:END:

A conflict occurs when Git cannot determine which version of a line is correct.

1. *Git pauses* and inserts conflict markers into the file.
2. *You edit* the file: choose what to keep or combine both versions.
3. =git add <file>= to mark it resolved.
4. =git commit= to finish the merge.

*** What Git Writes
:PROPERTIES:
:BEAMER_ENV: exampleblock
:BEAMER_ACT: <2->
:END:

#+BEGIN_SRC text
<<<<<<< HEAD
return total * 1.08   # your version
=======
return total * 1.10   # incoming version
>>>>>>> feature/tax-update
#+END_SRC

** Time to Explore: Branching and Merging

*** The Conflict Challenge
:PROPERTIES:
:BEAMER_ENV: block
:END:

1. =git init= a project. Create =notes.txt= and commit it on =main=.
2. =git switch -c feature=. Change line 1 of =notes.txt= and commit.
3. Switch back to =main=. Change line 1 to something different and commit.
4. Run =git merge feature=. Conflict!
5. Open =notes.txt=, fix the markers, =git add=, and =git commit=.
6. Run =git log --oneline --graph --all= to see the merge commit.

* Rebasing

** Rebase: Replaying on a New Base

*** The Idea
:PROPERTIES:
:BEAMER_ENV: block
:END:

Rebase takes your commits and replays them on top of another branch, producing a linear history.

#+BEGIN_SRC bash
# While on your feature branch:
git rebase main
#+END_SRC

Git re-applies your commits one by one onto =main='s latest commit and assigns them new hashes.

| =reword= | Keep the changes, but edit the commit message |
| =squash= | Combine this commit with the one above it |
| =drop= | Completely delete the commit and its changes |

- Use this to clean up "WIP" commits before opening a Pull Request.

** Discussion: Squash for PRs

*** Scenario
:PROPERTIES:
:BEAMER_ENV: block
:END:

You have 5 commits with messages like ="fix"=, ="typo"=, and ="asdf"=. You want to open a professional Pull Request.

*** Question (Discuss at your tables)
:PROPERTIES:
:BEAMER_ENV: exampleblock
:END:

How do you turn these into one clean commit?
What is the command and the rebase action you would use?

*** Answer
:PROPERTIES:
:BEAMER_ENV: alertblock
:BEAMER_ACT: <2->
:END:

Run =git rebase -i HEAD~5=. Keep the first commit as =pick= and mark the others as =squash=. This combines them all into one clean commit.

** Rebase Warning

*** The Rule
:PROPERTIES:
:BEAMER_ENV: alertblock
:END:

*Never rebase commits that have been pushed.*

Rebasing rewrites hashes. If others have based their work on your old hashes, you'll create a mess for everyone.

*** When to Use
:PROPERTIES:
:BEAMER_ENV: exampleblock
:BEAMER_ACT: <2->
:END:

Rebase is safe on local feature branches that haven't been pushed yet.

* Advanced Workflows

** Cherry-Pick

*** The Command
:PROPERTIES:
:BEAMER_ENV: block
:END:

Copies *one specific commit* from another branch and applies it to your current branch.

#+BEGIN_SRC bash
git cherry-pick <hash>
#+END_SRC

Creates a new commit with the same changes but a different hash.

*** When to Use It
:PROPERTIES:
:BEAMER_ENV: exampleblock
:BEAMER_ACT: <2->
:END:

| Scenario | Why Cherry-Pick? |
|----------|-----------------|
| Hotfix on =main= needed on =release= | Avoid merging all of =main= |
| One useful commit from an abandoned branch | No need to revive the whole branch |
| Backport a fix to an older version | Target exactly one change |

** Main Points

***                                                          :B_columns:
:PROPERTIES:
:BEAMER_ENV: columns
:END:

**** The Mental Model                                         :BMCOL:
:PROPERTIES:
:BEAMER_COL: 0.48
:END:

- Git stores *snapshots*, not diffs.
- Every commit points to its parent, forming a graph.
- Three stages: Working $\rightarrow$ Staging $\rightarrow$ Repo.
- Branches are cheap pointers.

**** Best Practices                                           :BMCOL:
:PROPERTIES:
:BEAMER_COL: 0.48
:END:

- Remotes are other copies of the same graph.
- =origin/main= is a local bookmark, not a live connection.
- Use rebase for linear history on local branches only.
- =reflog= and =revert= are your safety nets.
