* Git

* Learning Objectives

** Goals for Today
:PROPERTIES:
:BEAMER_ENV: block
:END:

By the end of this lecture, you will be able to:

- Model Git as a *graph of snapshots*, not a backup tool.
- Use the three-stage workflow: Working Directory $\rightarrow$ Staging $\rightarrow$ Repository.
- Create branches, merge changes, and resolve conflicts.
- Push and pull with remotes, and understand tracking branches.
- Recover from mistakes using =reflog= and =revert=.

* The Mental Model

** Why Git?

*** The Problem
:PROPERTIES:
:BEAMER_ENV: block
:END:

You are working on a project. Things go wrong.

- "It worked yesterday — what changed?"
- "I deleted that function and now I need it back."
- "My teammate's changes broke my code."

*** Git Solves This
:PROPERTIES:
:BEAMER_ENV: exampleblock
:BEAMER_ACT: <2->
:END:

Git is a *version control system*: a permanent, queryable record of every snapshot of your project, plus the tools to collaborate on it.

** Git is a Graph of Snapshots

*** The Commit
:PROPERTIES:
:BEAMER_ENV: block
:END:

Git thinks in *full snapshots* of the project, not diffs.

| Field | Description |
|-------|-------------|
| *Hash* | Unique ID (e.g. =a1b2c3d=) |
| *Parent(s)* | Link(s) back to previous snapshot |
| *Tree* | Pointer to the full file snapshot |
| *Message* | Human-readable description |

*** HEAD: You Are Here
:PROPERTIES:
:BEAMER_ENV: exampleblock
:BEAMER_ACT: <2->
:END:

=HEAD= is a pointer to your current position in the graph. It usually points to a branch, which points to a commit.

* The Three Stages

** The Three Areas
:PROPERTIES:
:BEAMER_ENV: block
:END:

#+BEGIN_EXPORT latex
\begin{center}
\begin{tikzpicture}[node distance=5cm, auto,
    box/.style={rectangle, draw, fill=blue!20, text width=5.5em, text centered, rounded corners, minimum height=3em},
    line/.style={draw, -latex', thick}]
    \node [box] (wd) {Working\\Directory};
    \node [box, right of=wd] (idx) {Staging\\Area};
    \node [box, right of=idx] (repo) {Repository};

    \path [line] (wd) -- node[above]{\texttt{git add}} (idx);
    \path [line] (idx) -- node[above]{\texttt{git commit}} (repo);
\end{tikzpicture}
\end{center}

- *Working Directory*: Your scratchpad — files on disk as you edit them.
- *Staging Area (Index)*: The draft commit. You choose exactly what goes in.
- *Repository*: Permanent history — sealed commits in the graph.

** The Core Workflow
:PROPERTIES:
:BEAMER_ENV: exampleblock
:END:

#+BEGIN_SRC bash
$ git status              # Where am I? What's changed?
$ git diff                # Working vs. Staging
$ git diff --staged       # Staging vs. last commit
$ git add file.txt        # Stage a file
$ git add -p              # Stage interactively (hunk by hunk)
$ git commit -m "message" # Seal staging into a commit
$ git log --oneline --graph  # See the graph
#+END_SRC

** Think-Pair-Share: What Gets Committed?

*** The Scenario
:PROPERTIES:
:BEAMER_ENV: block
:END:

You edit =app.py= and =README.md=. You run =git add app.py= then =git commit -m "fix login bug"=.

*** The Question
:PROPERTIES:
:BEAMER_ENV: exampleblock
:END:

What is in the commit? What is *not* in the commit?

*** Answer
:PROPERTIES:
:BEAMER_ENV: alertblock
:BEAMER_ACT: <2->
:END:

Only =app.py= is committed. The changes to =README.md= are still in the Working Directory — unstaged. The staging area is your *explicit choice* of what to snapshot.

** Writing Good Commit Messages

***                                                          :B_columns:
:PROPERTIES:
:BEAMER_ENV: columns
:END:

**** Convention                                               :BMCOL:
:PROPERTIES:
:BEAMER_COL: 0.48
:END:

- Short imperative subject line (≤50 chars)
- Blank line, then optional longer body
- Explain *why*, not just *what*

#+BEGIN_SRC text
Add login rate limiting

Prevents brute-force attacks
by capping attempts per IP.
Fixes #142.
#+END_SRC

**** Anti-Patterns                                            :BMCOL:
:PROPERTIES:
:BEAMER_COL: 0.48
:END:

| Bad Message | Problem |
|-------------|---------|
| ="stuff"= | Meaningless |
| ="Updated file.txt"= | What changed? |
| ="Fixed bug"= | Which bug? |
| ="WIP"= | Useless in log |

** The =.gitignore= File

*** Ignored Files
:PROPERTIES:
:BEAMER_ENV: block
:END:

Tells Git which files to *never* track.

| Pattern | What It Ignores |
|---------|-----------------|
| =*.log= | All log files |
| =node_modules/= | Entire directory |
| =.env= | Secrets file |
| =.DS_Store= | macOS metadata junk |
| =__pycache__/= | Python bytecode |

*** Rule of Thumb
:PROPERTIES:
:BEAMER_ENV: alertblock
:BEAMER_ACT: <2->
:END:

Create =.gitignore= *early*. Removing already-tracked files is more annoying than ignoring them from the start.

* Branching

** Branches Are Cheap Pointers

*** What Is a Branch?
:PROPERTIES:
:BEAMER_ENV: block
:END:

A branch is a *lightweight pointer* to a commit. Creating one does not copy any files.

*** Why Branch?
:PROPERTIES:
:BEAMER_ENV: exampleblock
:BEAMER_ACT: <2->
:END:

- Isolate risky experiments from =main=
- Work on multiple features simultaneously
- Code review: a branch is a unit of reviewable work

** Branch Commands
:PROPERTIES:
:BEAMER_ENV: block
:END:

| Command | Action |
|---------|--------|
| =git branch <name>= | Create a new pointer at HEAD |
| =git switch <name>= | Move HEAD to that branch |
| =git switch -c <name>= | Create *and* switch in one step |
| =git branch -d <name>= | Delete a branch pointer |
| =git branch -v= | List branches with latest commit |

* Merging & Conflicts

** Fast-Forward Merge

*** How It Works
:PROPERTIES:
:BEAMER_ENV: block
:END:

- =main= has not moved since the branch was created
- Git slides the pointer forward — *no new commit*
- Linear history preserved

:PROPERTIES:
:BEAMER_ENV: exampleblock

** Merge Commit

*** How It Works
:PROPERTIES:
:BEAMER_ENV: block
:END:

- Both branches have new commits (diverged)
- Git creates a new commit with *two parents*
- Full branch history is preserved

*** Command \& Result
:PROPERTIES:
:BEAMER_ENV: exampleblock
:BEAMER_ACT: <2->

** Conflict Resolution Workflow
:PROPERTIES:
:BEAMER_ENV: block
:END:

A conflict occurs when Git cannot mathematically decide which line is correct.

1. *Git pauses* and inserts conflict markers into the file.
2. *Human edits* the file — choose lines to keep, or combine both.
3. =git add <file>= — tell Git "I resolved it."
4. =git commit= — finalize the merge.

** Reading Conflict Markers
:PROPERTIES:
:BEAMER_ENV: exampleblock
:END:

#+BEGIN_SRC text
<<<<<<< HEAD (Current Change)
return total * 1.08   # your version
=======
return total * 1.10   # incoming version
>>>>>>> feature/tax-update (Incoming Change)
#+END_SRC

*** Key Point
:PROPERTIES:
:BEAMER_ENV: alertblock
:BEAMER_ACT: <2->
:END:

These markers are what Git *actually writes*. Editors like VS Code label them "Current" vs "Incoming" as a convenience layer on top.

*** Challenge
:PROPERTIES:

1. =git init= a new project and make two commits on =main=.
2. =git switch -c feature= and make one commit.
3. Switch back to =main=, edit the *same line* you changed on =feature=, and commit.

5. Run =git log --oneline --graph --all= to see the merge commit.

* Remotes

** Remotes: Just Another Graph Copy
:PROPERTIES:
:BEAMER_ENV: block
:END:

A *remote* is another copy of the repository on someone else's machine (or GitHub's servers). The graphs are synchronized by pushing and pulling commits.

| Term | Meaning |
|------|---------|
| =origin= | Default nickname for the remote URL |
| =upstream= | The original repo you forked from |
| =origin/main= | Your local *bookmark* of where the remote's branch was last checked |

** Push, Fetch, Pull

*** Commands
:PROPERTIES:
:BEAMER_ENV: block
:END:

| Command | What It Does |
|---------|--------------|
| =git push= | Upload local branch + commits to remote |
| =git push -u origin <branch>= | First push; links local branch to remote |
| =git fetch= | Download new commits; *do not* touch working files |
| =git pull= | =fetch= + =merge= in one step |

*** Common Scenario
:PROPERTIES:
:BEAMER_ENV: exampleblock
:BEAMER_ACT: <2->
:END:

*"Your branch is behind origin/main"* — the remote has commits you don't. Run =git pull= to catch up.

*** Note on =-u=
:PROPERTIES:

* Rebasing

** Rebase: Replaying on a New Base

*** The Goal
:PROPERTIES:
:BEAMER_ENV: block
:END:

*Linear history* — a clean, straight log instead of a web of merge commits.

*Metaphor*: "Pick up my commits and replay them on top of the new =main=."

*** Mechanics
:PROPERTIES:
:BEAMER_ENV: exampleblock
:BEAMER_ACT: <2->
:END:

#+BEGIN_SRC bash
# While on your feature branch:
git rebase main
#+END_SRC

Git re-applies your commits one by one on top of =main='s latest commit.

** The Golden Rule of Rebasing

*** Golden Rule
:PROPERTIES:
:BEAMER_ENV: alertblock
:END:

*Never rebase public history* — commits that have already been pushed and shared with others.

Rebasing rewrites commit hashes. If teammates have built on top of those commits, you create a divergence nightmare.

*** Safe Rebase Pattern
:PROPERTIES:
:BEAMER_ENV: exampleblock
:BEAMER_ACT: <2->
:END:

Rebase is safe when used *locally* on a feature branch you haven't pushed yet, or on a branch only you use.

* Cherry-Pick

** Cherry-Pick: The Surgical Tool

*** The Command
:PROPERTIES:
:BEAMER_ENV: block
:END:

Copies *one specific commit* from another branch and applies it to your current branch.

#+BEGIN_SRC bash
git cherry-pick <hash>
#+END_SRC

Creates a *new* commit with the same changes but a *different hash*.

*** When to Use It
:PROPERTIES:
:BEAMER_ENV: exampleblock
:BEAMER_ACT: <2->
:END:

| Scenario | Why Cherry-Pick? |
|----------|-----------------|
| Hotfix landed on =main=, need it on =release= | Avoid merging all of =main= |
| Pull one useful commit from an abandoned branch | No need to revive the whole branch |
| Backport a fix to an older version | Target exactly one change |

* Safety Nets

** The Reflog: Git's Undo Button
:PROPERTIES:
:BEAMER_ENV: exampleblock
:END:

Git records *every time HEAD moves* — even destructive operations like =reset --hard=.

#+BEGIN_SRC bash
# Oops — destroyed work with a hard reset
$ git reset --hard HEAD~2

# Find the lost commit in reflog
$ git reflog
a1b2c3d HEAD@{2}: commit: Add payment feature  <- this one!

# Restore it
$ git reset --hard a1b2c3d
#+END_SRC

** Revert: The Safe Undo

*** =git revert <hash>=
:PROPERTIES:
:BEAMER_ENV: block
:END:

Creates a *new commit* that is the exact inverse of the target. History is never rewritten.

***                                                          :B_columns:
:PROPERTIES:
:BEAMER_ENV: columns
:END:

**** git reset                                                :BMCOL:
:PROPERTIES:
:BEAMER_COL: 0.48
:END:

- Moves the branch pointer backward
- *Rewrites history*
- *Unsafe* on public/shared branches
- Use locally only

**** git revert                                               :BMCOL:
:PROPERTIES:
:BEAMER_COL: 0.48
:END:

- Creates a new "undo" commit
- *Preserves history*
- *Safe* for public branches
- Use when others have your commits

** Think-Pair-Share: Reset or Revert?

*** The Scenario
:PROPERTIES:
:BEAMER_ENV: block
:END:

You pushed a commit to =main= that broke the build. Your teammate has already pulled it. You need to undo the damage.

*** The Question
:PROPERTIES:
:BEAMER_ENV: exampleblock
:END:

Do you use =git reset= or =git revert=? Why?

*** Answer
:PROPERTIES:
:BEAMER_ENV: alertblock
:BEAMER_ACT: <2->
:END:

*=git revert=.* Your teammate already has the commit — resetting would rewrite shared history and cause a painful divergence. Revert adds a safe "undo" commit on top without disturbing existing history.

** Safety Net Reference
:PROPERTIES:
:BEAMER_ENV: block
:END:

| Situation | Tool | Command |
|-----------|------|---------|
| Accidentally deleted a branch | Reflog | =git reflog= + =git switch -c <b> <hash>= |
| Pushed a bad commit to shared branch | Revert | =git revert <hash>= |
| Local commit is wrong, not pushed | Reset | =git reset --hard HEAD~1= |
| Want to undo a commit, keep changes staged | Reset (soft) | =git reset --soft HEAD~1= |

*** Challenge
:PROPERTIES:

* Summary

** Choosing the Right Workflow
:PROPERTIES:
:BEAMER_ENV: block
:END:

| Goal | Tool |
|------|------|
| Save progress locally | =add= + =commit= |
| Share work with team | =push= / =pull= |
| Isolate new feature | =switch -c <branch>= |
| Incorporate teammate's work | =merge= or =rebase= |
| Pull one specific fix | =cherry-pick= |
| Undo a shared commit | =revert= |
| Recover "lost" work | =reflog= |
