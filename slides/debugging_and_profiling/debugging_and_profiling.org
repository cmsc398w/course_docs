* Diagnostic Mindset

** The Core Problem
:PROPERTIES:
:BEAMER_ENV: block
:END:

Software does *exactly* what you tell it, not what you *intend*.

- Debugging = bridging the gap between intended vs. actual behavior
- Though time-intensive, systematic debugging saves time in the long run
- Instead of a reactive slowdown, structured debugging accelerates development

** A Systematic Approach: The Scientific Method

Debugging is *methodical*, not trial-and-error. Avoid "shotgun debugging" (changing things randomly until it works).

1. *Reproduce* $\rightarrow$ If you can't reproduce, you can't fix
2. *Isolate* $\rightarrow$ Use binary search strategy in code
3. *Form a hypothesis* $\rightarrow$ Develop a theory based on symptoms
4. *Test the hypothesis* $\rightarrow$ Only change one thing at a time
5. *Fix and verify* $\rightarrow$ Confirm it solves the issue

** The Psychological Tool: Rubber Duck Debugging
:PROPERTIES:
:BEAMER_ENV: block
:END:

/The practice of explaining your code, line-by-line, to an inanimate object./

- Forces you to slow down and articulate your logic.
- Often, you find the bug before finishing the explanation.
- *Why it works*: shifts brain from "creation" mode to "explanation" mode.

** Isolation Strategies: "Divide and Conquer"
:PROPERTIES:
:BEAMER_ENV: exampleblock
:BEAMER_ACT: <2->
:END:

How to find a bug in 10,000 lines of code?

1. *Binary Search the Code*: Comment out half the code. Does it still crash?
   - Yes $\rightarrow$ Bug is in the active half.
   - No $\rightarrow$ Bug is in the commented half.
2. *Git Bisect*: Binary search through commit history to find *when* the bug was introduced.
   - Good commit vs. Bad commit.

* Printf Debugging & Logging

** Print Debugging: The Foundation
:PROPERTIES:
:BEAMER_ENV: block
:END:

Brian Kernighan (1979): /"The most effective debugging tool is still careful thought, coupled with judiciously placed print statements."/

- Print statements = simplest debugging method
- Iterate by adding prints near problem areas until root cause is clear
- Preferred because it's quick, easy, and effective

** Logging: Enhanced Print Debugging
:PROPERTIES:
:BEAMER_ENV: exampleblock
:BEAMER_ACT: <2->
:END:

Logging advantages:

- Output to files, sockets, remote servers (easier to review than terminal)
- Severity levels: =INFO=, =DEBUG=, =WARN=, =ERROR=
- Long-term monitoring + immediate debugging
- Structured logging (key–value pairs) $\rightarrow$ searchable, filterable, analyzable

** Logging Best Practices
:PROPERTIES:
:BEAMER_ENV: alertblock
:BEAMER_ACT: <3->
:END:

| Practice | Purpose |
|----------|---------|
| Set log levels properly | Filter noise |
| Use structured logging | Key–value pairs for analysis |
| Ensure traceability | Unique messages/prefixes to source |
| Use correlation IDs | Track requests end-to-end |
| Be strategic | Don't let logging consume resources |
| *Never log sensitive info* | Security first |

** Good vs. Bad Logging
:PROPERTIES:
:BEAMER_ENV: block
:END:

| Bad Log Message | Good Log Message | Why? |
|-----------------|------------------|------|
| "Error happened" | "Failed to upload file: timeout" | Specificity |
| "User 123" | "User 123 logged in from IP..." | Context |
| "Catch block reached" | "Database connection failed: retrying..." | Intent/Action |
| "Debug: 5" | "Calculated tax: 5.00 for order 99" | Meaning |

** Summary: Choosing the Right Tool
:PROPERTIES:
:BEAMER_ENV: block
:END:

| Tool | Best For... | Cons |
|------|-------------|------|
| *Printf / Logs* | Flow tracing, production issues, long-running processes | Clutters code, requires re-running |

* Profiling

** Beyond Functional Correctness
:PROPERTIES:
:BEAMER_ENV: block
:END:

Even if your code behaves correctly, it might not be good enough if it takes all your CPU or memory.

- Algorithms classes teach big $O$ notation but not how to find hot spots
- *Premature optimization is the root of all evil*
- Profilers help you understand which parts take most time/resources
- Focus optimization efforts where they matter most

* Profiling via Internal Timing

** Types of Timing
:PROPERTIES:
:BEAMER_ENV: block
:END:

When measuring program execution, there are different types of time:

| Type | Description |
|------|-------------|
| *Real Time* | Total time from start to finish (wall clock) |
| *User Time* | CPU time spent running your program's code |
| *Sys Time* | CPU time spent on system-level tasks (OS calls, I/O) |

** Understanding the Differences
:PROPERTIES:
:BEAMER_ENV: exampleblock
:BEAMER_ACT: <2->
:END:

Wall clock time can be misleading:

- Your computer may run other processes simultaneously
- Program may wait for external events (network, disk I/O)
- Real Time = User Time + Sys Time + Wait Time

** Key Insight
:PROPERTIES:
:BEAMER_ENV: alertblock
:BEAMER_ACT: <3->
:END:

If your program spends lots of time waiting for external resources:

- Improve by reducing wait times, not making code faster
- Send fewer requests to servers
- Cache information to avoid re-checking files on disk

** Python Example: =cProfile=
:PROPERTIES:
:BEAMER_ENV: exampleblock
:BEAMER_ACT: <2->

#+BEGIN_SRC python
import cProfile
import re

def grep_python(pattern, filename):
    with open(filename, 'r') as f:
        for line in f:
            if re.search(pattern, line):
                print(line.strip())

# Profile the function
cProfile.run('grep_python("import", "script.py")')
#+END_SRC

* Memory Profilers

** Memory Profiling Overview
:PROPERTIES:
:BEAMER_ENV: block
:END:

Track, measure, and analyze memory usage to detect inefficiencies and prevent memory issues like leaks.

** Common Memory Antipatterns
:PROPERTIES:
:BEAMER_ENV: alertblock
:END:

Where does memory go?

- *The Global List*: Appending to a global list (cache) without ever removing.
- *Unclosed Resources*: Forgetting to close files or database connections.
- *Reference Cycles*: Objects A and B point to each other, preventing garbage collection.
- *Large Data Loading*: Loading a 5GB file into RAM instead of processing line-by-line.

** Types of Memory Profiling
:PROPERTIES:
:BEAMER_ENV: exampleblock
:BEAMER_ACT: <2->
:END:

| Type | Purpose |
|------|---------|
| *Heap Profiling* | Track dynamic memory allocations/deallocations |
| *Stack Profiling* | Monitor memory in call stack (local variables) |
| *Reference Counting* | Track how many references exist to each object |

** Optimization Strategy
:PROPERTIES:
:BEAMER_ENV: alertblock
:BEAMER_ACT: <3->
:END:

*Avoid unnecessary allocations*:

- Memory allocation takes time and resources
- Garbage collection cycles introduce overhead
- Minimize allocations by reusing memory
- Use memory pools or object pooling techniques
- Avoid repeated allocations/deallocations
